user: we need to decide between PostgreSQL and MongoDB for the session store. the sessions are short-lived (TTL 24h), schema is fixed, we need atomic upserts, and the team knows SQL. estimated 50k sessions/day.
assistant: For those constraints, PostgreSQL is the clearer choice. Fixed schema + team familiarity + atomic upserts is exactly what relational handles well. You'll want: a sessions table with session_id (UUID PK), user_id, data (jsonb), created_at, expires_at, and a partial index on `WHERE expires_at > now()` for the lookup hot path. Run a cron every hour to DELETE WHERE expires_at < now() — or use pg_partman if volume grows. MongoDB adds operational complexity without buying anything here.
user: agreed. what about connection pooling?
assistant: Use PgBouncer in transaction mode between your app and Postgres. Session-store queries are short (single upsert or select), so transaction pooling is fine and you'll get much higher connection density. Set pool_size to 2x your app instance count. Add a health-check query (SELECT 1) with a 5s timeout in your app's startup sequence so bad pool config fails fast.
user: perfect, let's go with postgres + pgbouncer
assistant: Good call. I'll note the decision: PostgreSQL for session store with PgBouncer transaction-mode pooling. Key rationale: fixed schema, team SQL familiarity, atomic upsert requirements, 50k sessions/day volume fits comfortably within Postgres. Rejected: MongoDB (no advantage, more ops overhead).
